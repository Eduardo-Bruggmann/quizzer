Resumo das mudanças realizadas na API (arquitetura, regras de negócio e autenticação)

1. Banco e repositories
- Mantida a definição das tabelas em src/db/schema.ts (users, subjects, questions, alternatives) sem alterações estruturais, pois já estavam consistentes com os tipos.
- Ajustado src/repositories/userRepository.ts para:
  - createUser: passar a retornar o usuário criado (linha gerada por INSERT ... RETURNING).
  - findUserById e findUserByEmail: passar a retornar um único usuário (ou null) em vez de ignorar o resultado da query.
- Ajustado src/repositories/subjectRepository.ts, questionRepository.ts e alternativeRepository.ts para que as funções create* retornem a entidade criada usando RETURNING.
- Corrigido src/repositories/alternativeRepository.ts:
  - listAlternativesByQuestionId: removido uso incorreto de .all() e retornado diretamente o resultado do select do drizzle.

2. Services e regras de negócio
- src/services/userService.ts:
  - Garante que getUserById lança erro "User not found" se o usuário não existir.
  - Mantida a lógica de updateUser, updateUserRole, updateUserPassword e deleteUser usando o repository, sempre validando existência antes.
  - Corrigida duplicação acidental de getAllUsers, mantendo apenas uma definição.
- src/services/subjectService.ts, questionService.ts e alternativeService.ts:
  - Mantidas como camada fina de orquestração entre rotas e repositories, pois já estavam coerentes com a arquitetura proposta.
- src/services/utils.ts:
  - Mantida a função parseOrThrow e helpers parseUserRole/toSafeUser para centralizar validação e exposição segura de usuário.

3. Autenticação (sign up, sign in, tokens)
- src/lib/jwt.ts:
  - Mantida a validação forte de JWT_SECRET (obrigatório e com pelo menos 32 caracteres), além de issuer, audience e tempo de expiração configuráveis por env (JWT_ISSUER, JWT_AUDIENCE, JWT_ACCESS_EXPIRATION).
  - Funções createAccessToken e verifyAccessToken usadas como única forma de gerar e validar tokens, com payload contendo sub, email e role.
- src/schemas/authSchemas.ts e src/services/utils.ts:
  - Mantidos e utilizados para validação de entrada (Zod) e normalização de email/senha/role, garantindo regras de negócio claras na autenticação.
- src/services/authService.ts:
  - Renomeado o fluxo principal de cadastro para registerUser(data: RegisterData), mantendo as validações de entrada via Zod, checagem de usuário existente, hash de senha com bcrypt e criação do usuário via repository.
  - Adicionado alias export const signUp = registerUser para manter a semântica de "sign up" caso seja útil em outros pontos.
  - Mantidas e utilizadas as funções signIn (login usuário + geração de token) e forgotPassword (alteração de senha com validação de não reutilização de senha antiga).
- Rotas de autenticação Next.js (App Router):
  - src/app/api/auth/login/route.ts: continua usando signIn do authService, retornando { success: true, user, accessToken } em caso de sucesso e status 401 em caso de falha.
  - src/app/api/auth/register/route.ts: corrigido para usar a nova assinatura de registerUser, passando um objeto { fullName, email, password, confirmPassword } em vez de parâmetros soltos.

4. Rotas de API (CRUD) para usuários, matérias, questões e alternativas
- Usuários
  - [src/app/api/users/route.ts]:
    - GET /api/users: lista todos os usuários.
    - Suporte a filtro por role via query string: ?role=student ou ?role=teacher, usando userService.getStudents/getTeachers.
  - [src/app/api/users/[id]/route.ts]:
    - GET /api/users/:id: busca usuário por id via userService.getUserById, retornando 404 se não encontrado.
    - PATCH /api/users/:id: atualiza fullName e/ou email via userService.updateUser, retornando erro adequado em caso de usuário inexistente ou input inválido.
    - DELETE /api/users/:id: remove usuário via userService.deleteUser, com 404 se o usuário não existir.
  - Observação: a criação de usuário ("criar usuário") é feita pelo fluxo de registro em /api/auth/register, que já aplica todas as regras de validação e hashing de senha.

- Matérias (subjects)
  - [src/app/api/subjects/route.ts]:
    - GET /api/subjects: lista todas as matérias via subjectService.getAllSubjects.
    - POST /api/subjects: cria uma nova matéria gerando id com uuid, recebendo name, description e createdBy do corpo da requisição, e retornando a matéria criada.
  - [src/app/api/subjects/[id]/route.ts]:
    - PATCH /api/subjects/:id: atualiza name e/ou description via subjectService.updateSubject.
    - DELETE /api/subjects/:id: remove a matéria via subjectService.deleteSubject.

- Questões (questions)
  - [src/app/api/questions/route.ts]:
    - GET /api/questions?subjectId=...: lista questões filtradas por subjectId obrigatório, via questionService.getQuestionsBySubject.
    - POST /api/questions: cria nova questão gerando id com uuid, recebendo subjectId, statement e createdBy, retornando a questão criada.
  - [src/app/api/questions/[id]/route.ts]:
    - PATCH /api/questions/:id: atualiza o enunciado (statement) via questionService.updateQuestion.
    - DELETE /api/questions/:id: remove a questão via questionService.deleteQuestion.

- Alternativas (alternatives)
  - [src/app/api/alternatives/route.ts]:
    - GET /api/alternatives?questionId=...: lista alternativas de uma questão específica (questionId obrigatório) via alternativeService.getAlternativesByQuestion.
    - POST /api/alternatives: cria nova alternativa gerando id com uuid, recebendo questionId, content e isCorrect e retornando a alternativa criada.
  - [src/app/api/alternatives/[id]/route.ts]:
    - PATCH /api/alternatives/:id: atualiza content e/ou isCorrect via alternativeService.updateAlternative.
    - DELETE /api/alternatives/:id: remove alternativa via alternativeService.deleteAlternative.

5. Validações e testes automáticos
- Rodado o verificador de erros do TypeScript/Next (get_errors) em todo o workspace, sem erros de compilação reportados após as mudanças.
- Tentativa de executar npm install e npm run build diretamente no ambiente; npm install retornou código 1 (sem detalhes disponíveis via ferramenta), então não foi possível validar o build final por comando, mas o código compila do ponto de vista de tipos.

6. Observações de arquitetura e clareza
- Mantida a arquitetura em camadas: rotas -> services -> repositories -> db.
- Repositories agora sempre retornam dados de forma explícita (evitando funções que apenas executam queries sem retorno), o que simplifica regras de negócio nos services e garante clareza.
- Services concentram validações de negócio simples (existência de usuário, hash de senha, regras de senha, etc.) e delegam I/O ao repository.
- Rotas da API estão curtas, legíveis e focadas em:
  - ler/parsing do corpo ou query string;
  - chamar o service correspondente;
  - transformar a resposta em JSON + status HTTP adequado.
- A validação de JWT_SECRET continua centralizada em src/lib/jwt.ts, com mensagem clara caso a variável de ambiente esteja ausente ou fraca.
